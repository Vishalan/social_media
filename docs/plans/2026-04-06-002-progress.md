## Dispatch rules

1. **One subagent per unit at a time.** Two parallel agents on the same unit corrupt each other's work.
2. **Respect the dependency graph.** Before marking a unit "in progress", confirm all its listed dependencies are ✅.
3. **Use git worktrees for parallel waves C and E.** Both units in those waves touch `sidecar/db.py` schema — isolate with worktrees to avoid merge conflicts, then land them sequentially via the orchestrator.
4. **The orchestrator (main conversation) owns commits.** Subagents never commit. Subagents return a report; the orchestrator verifies and uses `cc-safe-commit` to land the work.
5. **Mark this tracker before and after each dispatch.** "Before" = set status to 🟡 with assignee name + timestamp. "After" = set to ✅ (if tests pass) or 🔴 with blocker text.
6. **If the plan file changes** (e.g., a new deferred question becomes critical), pause all active subagents and update the plan first. Subagents work from a snapshot — plan drift breaks them.
7. **Never skip Unit 1.** Even though it's mostly ops + docs, later units assume `deploy/portainer/docker-compose.yml` exists as their deployment target.
