# feat: Engagement Layer v2

## Overview

Three coordinated waves of engagement upgrades on top of the existing CommonCreed pipeline:

- **Tier A (engagement floor)** — phone-mockup synced article highlight, animated per-word burned-in captions, zoom-punch on keywords, and an in-repo SFX library auto-placed at every cut.
- **Tier B (differentiator formats)** — tweet/X-post reveal, A/B split-screen comparison, and denser editing-rhythm rules in the Haiku timeline planner.
- **Tier C (cinematic data viz)** — a new Remotion sidecar container exposing one HTTP endpoint, used by a `cinematic_chart` b-roll type for animated bar charts, number tickers, and line charts.

Tier 0 (shared substrate) ships first and unblocks all three tiers. Tiers A and B share a registration / branding surface; Tier C is independent and may run in parallel.

## Problem Frame

The existing pipeline's seven b-roll generators communicate "here is the article" but rarely deliver the goosebump moment of "the words I am hearing right now are highlighted on the screen as I read them." Burned-in captions render as static blocks. Tweet quotes and A/B comparisons collapse into generic browser visits. Animated charts don't exist. The work in this plan upgrades the engagement layer using the existing audio + script + word-timestamp substrate — no GPU-side changes, no avatar changes, no new stock-clip mining.

## Requirements Trace

- **R1.** Phone-mockup synced article highlight as a new b-roll type (`phone_highlight`).
- **R2.** Animated word-level burned-in captions on every short.
- **R3.** Zoom-punch on keyword segments (1.05–1.20× scale snap, 3–6 frames) timed to high-value tokens.
- **R4.** Vetted in-repo SFX library, auto-placed at cuts / reveals / keyword-punches.
- **R5.** Tweet/X-post reveal as a new b-roll type (`tweet_reveal`).
- **R6.** A/B split-screen comparison as a new b-roll type (`split_screen`).
- **R7.** Editing-rhythm rules added to the Haiku timeline planner (cut every 2–4 s, burst sequence every 15 s).
- **R8.** Remotion sidecar + `cinematic_chart` b-roll type for animated charts and diagrams.

## Scope Boundaries

(Inherited verbatim from the brainstorm; restated for plan-level clarity.)

- **No source-video clipping.** Phone-highlight in R1 renders cleaned article text in our own HTML template — never a screenshot of the live page. Real third-party footage is the deferred `source-clipping-pipeline` brainstorm.
- **No stylized animated hooks.** Ghibli/anime-style hooks live in the deferred `stylized-hook-visuals` brainstorm.
- **No avatar lip-sync or generation changes.** R2 captions are layered on top of the assembled video, not generated by the avatar.
- **No `caption_gen.py` (social caption / metadata) changes.** "Captions" in this plan = burned-in on-video subtitle text.
- **Existing b-roll generators keep working.** R1, R5, R6, plus C-tier `cinematic_chart` are net-new generators added to the catalog. The existing seven (`browser_visit`, `image_montage`, `code_walkthrough`, `stats_card`, `ai_video`, `emphasis_card`, `headline_burst`, `stock_video`) remain in place.
- **`VideoEditor.assemble()` contract is preserved.** It still takes one b-roll path per video. New optional kwargs (`keyword_punches`, `sfx_events`) are added; nothing existing is renamed.
- **No multi-segment composition refactor.** Internal segment timelines stay generator-internal, as today.
- **No WhisperX upgrade.** The existing faster-whisper word timestamps are accurate enough; WhisperX is a deferred follow-up.
- **Tier C may slip without blocking Tiers A and B.** If Remotion / Node container work proves materially harder than expected, Tier C ships separately or scopes down to "static animated chart PNG sequence rendered via Pillow" as fallback.

## Key Technical Decisions

The brainstorm's *Outstanding Questions → Deferred to Planning* section is resolved here. Each decision below corresponds to one of those questions.

- **R1 render strategy = PNG-per-timeline-event + FFmpeg concat.** Matches the existing `browser_visit.py` Playwright + concat pattern; supports auto-scroll between phrases without wrestling with CSS animation timing in headless Chromium. (CSS animation rejected: chromium audio sync drift in headless mode is unreliable.)
- **R1 article length handling = lead + 2 Haiku-picked paragraphs.** Articles >6 paragraphs get cropped to the lead plus the two most-relevant body paragraphs picked by the same Haiku call that produces the timeline plan. Matches our 30–60 s short format.
- **R2 = ASS karaoke, not FFmpeg `drawtext`.** libass is confirmed in the existing pipeline (`scripts/video_edit/video_editor.py:590` uses `-vf ass=`). ASS supports color tweens and per-word timing tags; `drawtext` requires per-word filter chains that don't compose cleanly.
- **R3 zoom-punch insertion point = single combined FFmpeg final-pass inside `_assemble_broll_body`.** Replaces the existing two-pass `moviepy → ffmpeg ass=` flow with `moviepy → single ffmpeg pass that does (zoompan) + (ass burn) + (sfx mix) in one filter graph`. One re-encode instead of three; honors the unified `final_make_frame` invariant by operating *after* the make-frame stage, not inside it.
- **R4 SFX pack = curated CC0 / commercial-permitted bundle of ~15 files in `assets/sfx/`.** Sources: Mixkit (free for commercial use), Pixabay (CC0). Each file <50 KB, <500 ms. License manifest committed alongside in `assets/sfx/LICENSE.md`. No SFX-generation API; no per-video generation.
- **R5 tweet card style = CommonCreed-branded, not native X dark theme.** White card on navy `#1E3A8A` matte with sky-blue `#5C9BFF` verified badge and divider. Stays on-brand; sidesteps trademark risk.
- **R5 tweet detection = added to the existing topic-selection Haiku call**, returning a structured `tweet_quote` field when the source article quotes a named person.
- **R6 A/B split = composer pattern.** `SplitScreenGenerator` invokes existing generators (`browser_visit`, `image_montage`, `stats_card`) with a new `width_override=540` parameter, then composes via FFmpeg `hstack`. No new sub-fetchers.
- **R7 max segment budget = `target_duration_s / 1.5`.** A 60 s short maxes at 40 segments. Planner is asked to compact (not the renderer to truncate) when over.
- **R8 Remotion image base = `node:20-bullseye-slim` + chromium + ffmpeg + fontconfig.** Target image size ≤ 1.8 GB. Built from `deploy/remotion/Dockerfile`, deployed as a peer container on the existing `commoncreed` bridge network at `http://commoncreed_remotion:3030`.
- **R8 v1 chart templates = bar chart, number ticker, line chart (3 templates).** Sufficient for benchmark, model-size, and timeline-trend visuals respectively. Additional templates land in follow-up plans.
- **Brand assets centralized in `scripts/branding.py`.** Today every generator carries its own `_BOLD_CANDIDATES = [...]` list and per-generator color literals. Tier 0 introduces one module of truth; existing generators are migrated opportunistically (Tier 0 migrates `headline_burst.py` as a proof, the rest follow as touched).
- **Three-file registration linter test.** A pytest case asserts every type in `BrollSelector._VALID_TYPES` has a matching factory branch and is present in BOTH the `primary` and `fallback` enums of the JSON schema. Prevents silent partial registration.
- **Both `smoke_e2e.py` and `commoncreed_pipeline.py` updated for every pipeline-flow change.** smoke_e2e is what runs in production; commoncreed_pipeline is the canonical reference. The flow change for R3 (keyword-punch extraction between transcribe and assemble) ships in both.
- **Stdout contract preserved.** The final `print(...)` block in `smoke_e2e.py` parsed by `sidecar/pipeline_runner.py` is extended additively (new keys appended), never restructured.
- **Trimmed-audio clock is the single master.** `caption_segments`, `keyword_punches`, and SFX events are all expressed in trimmed-audio time and consumed downstream without re-resolution. This eliminates the multi-clock drift surfaced in flow analysis.

### Tier A — Engagement Floor

#### Unit A1 — Phone-highlight generator (`phone_highlight`)

**Goal:** A new b-roll generator that renders a vertical phone mockup of the article body with the currently-spoken phrase highlighted in brand sky-blue, auto-scrolling so the active phrase sits in the upper third.

**Requirements:** R1.

**Dependencies:** Unit 0.1 (brand colors + Inter font), Unit 0.4 (article extract on `VideoJob`), Unit 0.5 (selector + registration).

**Files:**
- Create: `scripts/broll_gen/phone_highlight.py`
- Create: `assets/templates/phone_article.html.j2` (Jinja2 template)
- Create: `assets/templates/iphone_chrome.png` (1080×1920 device frame overlay PNG, transparent background)
- Create: `scripts/broll_gen/tests/test_phone_highlight.py`
- Modify: `scripts/broll_gen/factory.py` (replace placeholder branch with real generator)

**Approach:**

Phrase-level chunking (decided in brainstorm):
- Group `caption_segments` into phrases of 3–7 words.
- Phrase boundary = punctuation in script (`,`, `.`, `?`, `!`, `;`, `:`) OR a 250+ ms silence gap between consecutive word timestamps OR conjunctions at boundaries (`and`, `but`, `or`, `so`).
- Each phrase carries `{text, t_start, t_end, source_paragraph_index}`.

Article-text trim (Haiku call):
- Input: `extracted_article` + script text.
- Prompt: "Pick the lead paragraph plus the two body paragraphs most relevant to the script. Return their indices."
- Output: `{lead_index, picked_indices: [int, int]}`.
- Resulting article view = `[lead] + [picked paragraph 1] + [picked paragraph 2]`. Capped at 6 paragraphs even if all picked.

Phrase-to-paragraph alignment:
- For each phrase, find the longest substring match against the trimmed article text. Phrases that don't match are rendered as "spoken" but unhighlighted (script paraphrasing is normal).
- Match success rate logged: warn if <60% of phrases match (signals bad article extraction or heavy script rephrasing).

Render strategy (decided: PNG-per-event + concat):
- For each phrase event:
  1. Render `phone_article.html.j2` with the matched span wrapped in `<mark>`, all earlier matched spans wrapped in `<mark class="past">` (lighter sky-blue), later spans plain.
  2. Compute scroll offset so the `<mark>` is at 33% of viewport height.
  3. Playwright screenshot at 1080×1920.
- Concat the PNGs via FFmpeg `concat` demuxer with 200 ms `xfade=transition=fade` between events.
- Composite the iPhone chrome PNG overlay via `overlay` filter.

CSS for the highlight (uses `box-decoration-break: clone` for clean multi-line wrap):
```css
mark {
  background: #5C9BFF;
  color: #FFFFFF;
  box-decoration-break: clone;
  -webkit-box-decoration-break: clone;
  padding: 0.1em 0.2em;
  border-radius: 0.2em;
}
mark.past { background: #5C9BFF55; color: #1E3A8A; }
```

Playwright lifecycle (matches `browser_visit.py:418-567`):
- `async with async_playwright() as p` per call (no pool).
- `browser.new_context(**playwright.devices["iPhone 14 Pro"])` to get device emulation, then override viewport to 1080×1920 for rendering at our target resolution.

**Patterns to follow:**
- `scripts/broll_gen/browser_visit.py` Playwright + Haiku timeline + concat structure (the closest cousin).
- `scripts/broll_gen/headline_burst.py` PIL frame + concat-demuxer pattern (for the FFmpeg side).

**Verification:**
- Pytest: happy path (mocked Playwright + mocked Haiku) returns `output_path`.
- Pytest: missing `extracted_article` raises `BrollError("phone_highlight requires extracted article text")`.
- Pytest: factory registration test passes (already covered by Unit 0.6 linter).
- Smoke run: pick a real article topic, run end-to-end, **visually inspect** that phrase-level highlight tracks voiceover at the phrase level. Specifically: when the voiceover speaks "GPT-5 just launched", that exact phrase is the active highlighted region on screen at the same moment.

#### Unit A2 — Animated word-level ASS captions

**Goal:** Replace today's phrase-level static ASS captions with per-word animated captions. Active word gets a sky-blue background; inactive words are white with a navy drop-shadow. Applied uniformly across all assembly modes.

**Requirements:** R2.

**Dependencies:** Unit 0.1 (brand colors + ASS color helper), Unit 0.2 (libass smoke gate).

**Approach:**

Replace the current phrase-chunking logic in `_build_ass_captions` with per-word `Dialogue:` lines using ASS karaoke timing tags:

```
Dialogue: 0,0:00:01.20,0:00:01.45,Caption,,0,0,0,,{\1c&HFFFFFF&\3c&HBA3A1E&\bord3}GPT
Dialogue: 0,0:00:01.45,0:00:01.85,Caption,,0,0,0,,{\1c&HFFFFFF&\3c&HFF9B5C&\bord12}5
```

- Per-word timing comes directly from `caption_segments[i]['start']` and `['end']` (already the trimmed-audio clock — see Tier 0 invariant).

#### Unit A3 — Zoom-punch + SFX combined final-pass

**Goal:** Single FFmpeg final-pass that simultaneously applies (a) zoom-punch on keyword timestamps, (b) ASS caption burn-in, (c) SFX mix into the audio track. One re-encode instead of three. Honors the unified `final_make_frame` invariant by operating *after* the make-frame stage, never inside it.

**Requirements:** R3, R4.

**Dependencies:** Unit 0.3 (SFX library), Unit A2 (ASS captions).

**Files:**
- Modify: `scripts/video_edit/video_editor.py` (`_assemble_broll_body` at line 267; replace `_write_with_captions` with `_apply_engagement_pass`)
- Modify: `scripts/commoncreed_pipeline.py` (add keyword-punch extraction step between transcribe and assemble)
- Modify: `scripts/smoke_e2e.py` (mirror)
- Create: `scripts/content_gen/keyword_extractor.py`

**Approach:**

Keyword extraction (Claude Haiku):
- New module `keyword_extractor.py` exports `extract_keyword_punches(script_text, caption_segments) -> list[KeywordPunch]`.
- Prompt: "Identify the 4–7 highest-value tokens in this script that deserve a visual emphasis punch. Prefer proper nouns (model names, company names), dollar figures, percentages, and version numbers. Return JSON list of `{word, intensity}` where intensity ∈ {light, medium, heavy}."
- **Drift guard (critical):** for each emitted keyword, find the matching `caption_segments` entry by word text + scan the timestamps. If no match within ±100 ms tolerance, drop the keyword silently. Log per-drop.
- Returns `[{word, t_start, t_end, intensity}]` keyed in trimmed-audio time.

Pipeline ordering update:
- smoke_e2e.py: between `transcribe` (step 6) and `step_assemble` (step 7), call `extract_keyword_punches(...)`. Pass the result to `editor.assemble(..., keyword_punches=keyword_punches, sfx_events=sfx_events)`.
- commoncreed_pipeline.py: same insertion point in `_finalize_job` (line 446 area).

SFX event derivation:
- For every cut in the b-roll timeline (boundary between two `browser_visit` segments, etc.) → `SfxEvent(t, "cut", "light")` using `cut_whoosh` / `cut_swoop` / `cut_swish` (rotated by seed).
- For every keyword punch → `SfxEvent(t_start, "punch", intensity)` using `pop_*` files.
- For tweet-reveal / split-screen reveals (Tier B) → `SfxEvent(t_reveal, "reveal", "heavy")` using `swipe_in` + `ding_clean`.

Combined final-pass (`_apply_engagement_pass`):
- Single ffmpeg invocation:
  ```
  ffmpeg -y \
    -i base_video.mp4 \
    -i voiceover.mp3 \
    -i sfx_track.wav \
    -filter_complex "
      [0:v]zoompan=z='if(...keyword timing...)':d=N:s=1080x1920[vz];
      [vz]ass={ass_path}[vout];
      [1:a][2:a]amix=inputs=2:duration=first:weights=1.0 0.18[aout]
    " \
    -map "[vout]" -map "[aout]" \
    -c:v libx264 -preset fast -crf 18 \
    -c:a aac -b:a 192k \
    {output_path}
  ```
- The zoom-punch curve per keyword: `if(between(in_t,t0,t0+0.2), 1.0+0.15*sin(PI*(in_t-t0)/0.2), 1.0)` — quadratic bell over 200 ms peaking at 1.15×.
- Generates an intermediate `sfx_track.wav` from `mix_sfx_into_audio` (Unit 0.3).

**Patterns to follow:**
- Existing `subprocess.run([FFMPEG, "-y", ...])` pattern.
- Existing `_write_with_captions` (line 590) — replaced wholesale by `_apply_engagement_pass`; the contract (input video → output video) is preserved so call sites don't change.

**Verification:**
